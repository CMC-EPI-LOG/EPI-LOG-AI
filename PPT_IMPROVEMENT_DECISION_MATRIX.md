## 1) 현재 개발 PPT 진단 (좋은 점 / 보완점)

### 1.1 좋은 점

- 문제 정의 -> 해결 구조 -> 아키텍처 -> 트러블슈팅 흐름은 이미 잘 잡혀 있음.
- BFF 분리, Decision Engine, RAG, 데이터 모델까지 개발 스토리 라인이 존재함.
- API 문서 슬라이드(16~17p)로 FE/BE 계약을 보여주는 구조가 있음.

### 1.2 멘토 피드백 기준 주요 갭

- `왜 이 스택인가`가 정성 설명 위주라서, 비교군/점수/가중치가 필요함.
- 트러블슈팅의 `기대값 -> 실측값 -> 개선폭`이 수치로 안 보임.
- 분리 구조의 안정성과 확장성이 그림에서 "느껴지도록" 시각 밀도 보강 필요.
- 각 결정의 기획적 배경(어떤 제약에서 나온 선택인지) 설명이 더 필요함.

## 3) 의사결정 매트릭스 (Weighted Scoring Model)

> 점수 체계: 1(낮음) ~ 5(높음)  
> 가중 점수 = `가중치 × (점수/5)`  
> 총점 100점

### 3.1 결정 A: Web + BFF 프레임워크

#### 기획적 배경

- 부모에게 빠른 의사결정 제공이 핵심이므로, FE와 API 오케스트레이션의 결합이 필요했다.
- 작은 팀에서 배포/운영 복잡도를 낮춰야 했다.
- 실패 상황에서도 UI가 끊기지 않도록 BFF 레이어에서 예외를 흡수해야 했다.

#### 비교군

- 옵션 1: `Next.js(App Router + Route Handler)`
- 옵션 2: `React(Vite) + Express`
- 옵션 3: `NestJS Full-stack`

## 4) 트러블슈팅: 기대값/달성값/레슨런 (발표용 템플릿 + 실측 반영)

## 4.1 Case A. 지역 오매칭 / 측정소 정합성

### 배경

- 중복 지명(중구/강서구 등)으로 타 시도 데이터 매칭 위험 존재.

### 기대값(목표)

- `crossSidoMismatchCount = 0`
- `degradedRatio <= 20%`
- 전국/부산 주요 구 fixture에서 지역 정합성 100%

### 실측값(2026-03-04)

- 스모크 2세트 합산 `33건`
- `crossSidoMismatchCount = 0`
- `degradedRatio = 0.00%`
- 상태 분포: `LIVE 33 / STATION_FALLBACK 0 / DEGRADED 0`

### 개선 해석

- 목표치를 충족했고, 정합성 문제는 현재 샘플에서는 재현되지 않음.
- 주소 문자열과 실제 측정소 이름이 다른 케이스도 모두 정상 해석됨(33/33).

### 레슨런

- "주소"와 "측정소"를 같은 개념으로 다루면 장애가 생긴다.
- 후보군 + 시도 정합성 검증 + 신뢰성 메타 노출이 함께 있어야 운영이 가능하다.

---

## 4.2 Case B. API 처리 지연 (순차 호출 vs 병렬 오케스트레이션)

### 배경

- 사용자 체감은 평균보다 tail latency(P90/P95)에 크게 영향받음.

### 기대값(목표)

- `Avg latency 20%+ 개선`
- `P50 latency 20%+ 개선`
- tail latency의 급격한 상승 구간 완화

### 실측값(2026-03-04, station=강남구, n=20)

- 비교군(순차 호출): `avg 1826ms`, `p50 1129ms`, `p95 9356ms`
- 개선군(BFF 병렬): `avg 1065ms`, `p50 824ms`, `p95 3406ms`
- 개선율: `avg 41.7%`, `p50 27.0%`

### 개선 해석

- 평균/중앙값 모두 목표치를 초과 달성.
- tail 구간도 큰 폭으로 안정화됨.

### 레슨런

- "성공/실패"만이 아니라 "지연 분포"를 봐야 체감 품질이 보인다.
- 병렬화 + BFF 흡수는 사용자 경험에 직접적으로 영향을 준다.

---

## 4.3 Case C. AI/외부 실패 내성 (Fallback)

### 배경

- AI 호출 실패/네트워크 실패 시 안내 중단은 제품 신뢰를 즉시 훼손함.

### 기대값(목표)

- AI 실패 상황에서도 사용자-facing 응답 지속
- 관련 핵심 테스트 100% 통과

### 실측값(2026-03-04, 핵심 테스트만 분리 실행)

- `daily-report.route + stationResolution + clothing-recommendation`:
- `20 / 20 테스트 통과 (100%)`
- clothing route는 AI 호출 실패 시 `fallback-bff` 응답 반환 검증 완료

### 개선 해석

- 실패 상황에서 "완전 실패" 대신 "안전한 축소 응답"으로 전환됨.

### 레슨런

- 생성형 기능은 항상 fallback UX를 먼저 설계해야 한다.
- 실패를 숨기지 말고, 서비스 계약(응답 shape) 유지에 집중해야 한다.
